## Goal runtime `/goal resume` must restart the turn with the stored goal objective, not the literal resume command

- Symptom:
  - Feishu `/goal resume` can reach the Codex goal-runtime path and successfully start a new app-server turn, but the assistant replies in natural language that it has no resume-goal tool and asks the user to continue manually.
- Root cause:
  - The `/goal resume` branch in `CliExecutorService.ExecuteCodexGoalRuntimeStreamAsync(...)` correctly reloaded the current goal and set its status back to `active`.
  - But when restarting the app-server turn, it passed the literal prompt `"/goal resume"` into `turn/start`.
  - Codex then interpreted that string as a normal user instruction rather than an internal runtime control action, so the model answered conversationally about lacking a resume tool.
- Working rule:
  - Goal-runtime control commands should be consumed by WebCode itself.
  - After a successful `/goal resume`, the subsequent `turn/start` prompt must use the stored `goal.Objective`, not the literal control command text.
- Current implementation:
  - Resume turn prompt selection now lives in [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:3180).
  - Regression coverage lives in [CliExecutorServiceTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CliExecutorServiceTests.cs:538).
- Search hints:
  - `GoalRuntimeCommandKind.Resume`
  - `StartedTurnPrompts`
  - `ExecuteStreamAsync_WhenGoalRuntimeResumes_UsesGoalObjectiveForTurnPrompt`

## Feishu goal status cards must preserve full multiline plain-text status output when Codex adapters see no structured events

- Symptom:
  - Clicking `/goal` in Feishu can return only the last line such as `Time used: 86 seconds`, even though WebCode formatted a multi-line goal status summary.
- Root cause:
  - Goal runtime status commands return plain text, not JSONL assistant events.
  - Feishu `codex` streaming paths still routed that output through adapter fallback extraction.
  - `ExtractFallbackOutput(...)` parsed line-by-line and kept overwriting `lastUsefulContent`, so multi-line plain text degraded to the final line.
  - A further nuance on the `codex` path is that `CodexAdapter.ParseOutputLine(...)` still wraps non-JSON text lines as `raw` or `error` events.
  - Those lines are adapter-readable, but they are not true structured JSONL assistant events, so treating every parsed `CliOutputEvent` as “structured output seen” still collapses plain multi-line status text to the last line.
- Working rule:
  - If adapter fallback sees no real structured JSON/JSONL output events at all, preserve the full plain-text payload via `FormatMarkdownOutput(fullOutput)` instead of collapsing to the last parsed line.
  - For this decision, plain text that the adapter mapped to `raw` or `error` from a non-JSON line must still count as plain text, not as structured output.
- Current implementation:
  - Card-action fallback now preserves full plain text in [FeishuCardActionService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuCardActionService.cs:2324).
  - Channel-side fallback now does the same in [FeishuChannelService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuChannelService.cs:1867).
  - Regression coverage lives in [FeishuCardActionServiceTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/FeishuCardActionServiceTests.cs:2108).
- Search hints:
  - `sawStructuredOutput`

## Codex app-server stdio transport must use UTF-8 without BOM on .NET

- Symptom:
  - Feishu `/goal` quick actions can reach WebCode and create the streaming card, but the Codex app-server immediately logs:
    - `Failed to deserialize JSONRPCMessage: expected value at line 1 column 1`
- Root cause:
  - `CodexAppServerSessionManager` launched `codex app-server` with `ProcessStartInfo.StandardInputEncoding = Encoding.UTF8`.
  - On .NET, that encoding emits a UTF-8 BOM on the first write to stdin.
  - Codex app-server expects newline-delimited JSON from the first byte, so the BOM makes the first JSON-RPC message fail at column 1.
- Confirmed behavior:
  - A minimal .NET repro against the installed `codex.exe app-server` fails with `Encoding.UTF8`.
  - The same repro succeeds when stdin uses `new UTF8Encoding(false)`.
- Working rule:
  - For Codex app-server stdio transport, always use UTF-8 without BOM for stdin.
  - Keep stdout/stderr on the same explicit no-BOM UTF-8 setting for consistency.
- Current implementation:
  - Transport encoding is centralized in [CodexAppServerSessionManager.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CodexAppServerSessionManager.cs:11).
  - Regression coverage lives in [CodexAppServerSessionManagerTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CodexAppServerSessionManagerTests.cs:1).
- Search hints:
  - `Failed to deserialize JSONRPCMessage`
  - `StandardInputEncoding`
  - `UTF8Encoding(false)`
  - `GetTransportEncoding`

## Resumed Codex goal threads can carry a lingering active turn that must be interrupted before starting a new one

- Symptom:
  - A Feishu `/goal` submit can restore an existing Codex thread successfully, but then fail immediately with:
    - `当前 app-server 运行中已有一个 turn，无法再启动新的 turn。`
- Root cause:
  - The restored thread may still have a server-side active turn from an earlier interrupted or long-running execution.
  - WebCode can set or clear the goal on that thread, but starting a new turn without first interrupting the lingering turn fails.
- Working rule:
  - For Codex goal runtime paths that start a fresh turn after `thread/goal/set`, catch the active-turn conflict once, interrupt the current turn, then retry `turn/start`.
  - Keep this retry narrow to the goal-runtime turn start path; do not broadly swallow unrelated `InvalidOperationException`s.
- Current implementation:

## Feishu goal quick submit should confirm before overwriting an active goal turn

- Symptom:
  - Users can submit a new `/goal` from the Feishu card while the current session still has an active goal execution, or while Codex app-server still tracks a lingering active turn from the same session.

- Working rule:
  - For `submit_goal_quick_input`, detect both visible Feishu-side active execution and Codex app-server `ActiveTurnId`.

## Codex app-server notifications must be normalized before Feishu streaming can render goal output correctly

- Symptom:
  - Feishu `/goal` can create the streaming card and start the Codex app-server turn, but the card body stays on the thinking state or shows no useful assistant output for a long time.
  - Logs show repeated CardKit `PUT` updates without matching visible reply content.
- Root cause:
  - Codex app-server emits JSON-RPC notifications such as `item/agentMessage/delta`, `item/completed`, and `item/commandExecution/outputDelta`.
  - The old bridge in `CodexAppServerSessionManager` stripped those notifications down to raw `delta` text before handing them to the Feishu streaming pipeline.
  - That lost the event type and the camelCase item metadata that the existing Codex adapter depends on, so Feishu could not reliably distinguish assistant messages from verbose tool output.
  - Forwarding raw command output deltas also risks bloating card updates with large shell output that should not become reply正文.
- Working rule:
  - For Codex app-server streaming, convert useful notifications into compact adapter-readable JSONL before writing them to `StreamOutputChunk`.
  - Normalize app-server item types such as `agentMessage` and `commandExecution` to the adapter schema (`agent_message`, `command_execution`).
  - Keep assistant message deltas, tool start/completion summaries, and turn lifecycle events.
  - Suppress verbose command output delta mirroring so Feishu cards do not treat shell output as assistant正文.
- Current implementation:
  - Notification normalization now lives in [CodexAppServerSessionManager.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CodexAppServerSessionManager.cs).
  - Regression coverage lives in [CodexAppServerSessionManagerTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CodexAppServerSessionManagerTests.cs).
- Search hints:
  - `item/agentMessage/delta`
  - `item/commandExecution/outputDelta`
  - `TryBuildCliOutputJsonl`
  - `agent_message`

## Feishu goal overwrite confirmation must consider active card-action streaming executions, not only channel-side execution state

- Symptom:
  - Submitting a new `/goal` from a Feishu card while that same card-action service still has an active streaming execution can skip the overwrite confirmation card and go straight into starting a new goal.
  - The old streaming card is cancelled first, then `CliExecutorService` later discovers the lingering app-server turn and interrupts it during retry, which looks like a silent overwrite to the user.
- Root cause:
  - `FeishuCardActionService.HasGoalExecutionConflict` originally checked only `_feishuChannel.IsSessionExecutionActive(sessionId)` and `ICodexAppServerSessionManager.HasActiveTurn(sessionId)`.
  - Card-action-triggered streaming executions are tracked separately inside `FeishuCardActionService._activeStreamingExecutions`, so that in-flight state was invisible to the conflict check.
  - In addition, `ICodexAppServerSessionManager` must be shared as a singleton; otherwise card actions and CLI execution can observe different app-server session state.
- Working rule:
  - Treat an active `FeishuCardActionService` streaming execution for the same session as a goal conflict.
  - Register `ICodexAppServerSessionManager` as a shared singleton so UI-side conflict checks and CLI-side goal runtime startup inspect the same `ActiveTurnId`.
- Search hints:

  - `HasGoalExecutionConflict`

## Goal runtime Feishu cards should expose only goal controls, and `pause_goal` must route through goal runtime instead of generic stop

- Symptom:
  - After enabling Codex app-server goal runtime, Feishu streaming cards still showed superpowers quick input, `继续`, plan buttons, and sometimes `停止`, which mixed one-shot workflow controls with goal-runtime controls.
  - In the same mode, clicking `/goal pause` could still follow the generic stop-streaming path instead of the unified goal command path.
- Root cause:
  - Both `FeishuCardActionService` and `FeishuChannelService` assembled bottom prompts/actions by always adding superpowers controls first, then goal controls, without checking `UseGoalRuntime`.
  - Card-action dispatch still special-cased `pause_goal` to `HandleStopStreamingExecutionAsync(...)` instead of the shared `HandleGoalQuickActionAsync(...)` path.
  - Goal-action busy gating treated every non-status action as blocked while Feishu marked the session busy, which would also suppress legitimate control actions like `pause` and `clear`.

    - the goal quick input
    - `/goal`
    - `/goal pause`
    - `/goal clear`
    - `/goal resume`
  - In that mode, hide superpowers quick input, `继续`, stop, and plan/subagent-plan actions.
  - Route `pause_goal` through the same goal quick action execution path as the other goal buttons so WebCode parses the command and uses the app-server goal/turn APIs.
  - Keep `pause` and `clear` exempt from the generic “session already running” guard; otherwise the controls are visible but unusable.
- Current implementation:

## Codex app-server session manager must use typed logger injection or WebCodeCli will fail at host startup

- Symptom:
  - `WebCodeCli.exe` can build successfully, but process startup fails during host construction with:
    - `Unable to resolve service for type 'Microsoft.Extensions.Logging.ILogger' while attempting to activate 'WebCodeCli.Domain.Domain.Service.CodexAppServerSessionManager'`
- Root cause:
  - `CodexAppServerSessionManager` constructor accepted non-generic `ILogger`.
  - Default ASP.NET Core / Extensions DI registers `ILogger<TCategoryName>`, not arbitrary bare `ILogger` constructor parameters.
  - `CliExecutorService` also had a manual fallback path that instantiated `new CodexAppServerSessionManager(_logger)`, which became invalid once the manager should use its own typed logger category.
- Working rule:
  - Services resolved from DI should request `ILogger<TService>` unless there is a very specific custom logger factory flow.
  - If a service has an out-of-container fallback `new` path, resolve the matching typed logger from `IServiceProvider`, and fall back to `NullLogger<T>.Instance` if needed.
- Current implementation:
  - Typed logger injection now lives in [CodexAppServerSessionManager.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CodexAppServerSessionManager.cs:20).
  - The manual fallback path now resolves `ILogger<CodexAppServerSessionManager>` in [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:72).
  - Regression coverage lives in [CodexAppServerSessionManagerTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CodexAppServerSessionManagerTests.cs:10).

## Concurrent goal-runtime requests for the same session must serialize app-server session creation

- Symptom:
  - A single visible Feishu `/goal` submission can log one `submit_goal_quick_input` callback, but `CodexAppServerSessionManager` still emits duplicated Node startup lines for the same `SessionId`.
  - One of those concurrent initializations can time out in `SendRequestAsync("initialize")` with `TaskCanceledException`, while the other app-server process comes up moments later and logs `Codex app-server 会话已启动`.
- Root cause:
  - `GetOrCreateSessionAsync` previously did an unlocked check-then-create sequence against `_sessions`.
  - Two concurrent callers for the same logical session could both observe “no running session” and race into `CreateSessionAsync`, launching two `codex app-server` processes for the same `sessionId`.
  - The loser could then surface as `启动 Codex app-server goal runtime 失败: A task was canceled.` even though the winner eventually initialized successfully.
- Working rule:
  - App-server session creation must be serialized per `sessionId`.
  - Guard the whole “check existing / cleanup stale / create new / store session” critical section with a keyed `SemaphoreSlim`.
- Current implementation:
  - Per-session creation locking now lives in [CodexAppServerSessionManager.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CodexAppServerSessionManager.cs:20).
  - Regression coverage lives in [CodexAppServerSessionManagerTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CodexAppServerSessionManagerTests.cs:145).
- Search hints:
  - `RunWithSessionCreationLockAsync`
  - `_sessionCreationLocks`
  - `GetOrCreateSessionAsync`
