## Feishu attachment submission card: input max length is capped by the platform

- Symptom:
  - The attachment reply card is created through CardKit, but Feishu returns business error `11310` when the card contains `form.input.max_length` above the supported default limit.
- Confirmed behavior:
  - For this form input path, Feishu rejected `max_length = 4000` with:
    - `ErrMsg: max_length exceed the default maximum 1000`
- Working rule:
  - Keep attachment supplemental text inputs at `1000` characters or below.
- Current implementation:
  - Shared constants live in [FeishuAttachmentSubmissionDefaults.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Model/FeishuAttachmentSubmissionDefaults.cs:1).
  - Card payload uses that limit in [FeishuAttachmentSubmissionCardHelper.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuAttachmentSubmissionCardHelper.cs:1).
  - Server-side submit validation also enforces it in [FeishuCardActionService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuCardActionService.cs:624).
- Search hints:
  - `max_length exceed the default maximum 1000`
  - `submit_attachment_prompt`
  - `feishu_attachment_prompt`

## Feishu form-submit buttons should follow the existing Superpowers pattern

- Symptom:
  - A reply attachment card can be created successfully, but clicking the submit button may not produce any `card.action.trigger` callback in WebCode logs.
- Likely cause:
  - The button shape drifted away from an already working Feishu form-submit pattern used elsewhere in the codebase.
- Working reference:
  - Reuse the same structure as the Superpowers bottom prompt submit button in [FeishuCardKitClient.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuCardKitClient.cs:687).
