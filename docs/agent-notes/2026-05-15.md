## Feishu attachment submission card: input max length is capped by the platform

- Symptom:
  - The attachment reply card is created through CardKit, but Feishu returns business error `11310` when the card contains `form.input.max_length` above the supported default limit.
- Confirmed behavior:
  - For this form input path, Feishu rejected `max_length = 4000` with:
    - `ErrMsg: max_length exceed the default maximum 1000`
- Working rule:
  - Keep attachment supplemental text inputs at `1000` characters or below.
- Current implementation:
  - Shared constants live in [FeishuAttachmentSubmissionDefaults.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Model/FeishuAttachmentSubmissionDefaults.cs:1).

## Feishu pasted inline images arrive as `post`, not top-level `image`

- Symptom:
  - A user pastes an image directly into Feishu's rich-text input box and sends text with it.
  - WebCode logs `MessageType=post`, and the normalized prompt becomes something like:
    - `[image]`
    - `这是啥？`
  - If this path is treated as plain text, CLI only sees the placeholder string and not the actual image bytes.
- Confirmed behavior:
  - This input does not use the same payload shape as top-level `image` messages.
  - The real image key is nested under rich-text nodes such as:
    - `content[][ { "tag": "img", "image_key": "..." } ]`
- Working rule:
  - Parse inline `img` nodes from Feishu `post` payloads into `FeishuIncomingMessage.Attachments`.
  - When a `post` message contains both inline image attachments and real text, bypass the attachment submission card and submit the text plus attachment directly to CLI through the prepared-submission path.
  - Keep the text prompt clean by stripping placeholder markers like `[image]` and `[media]` before building the CLI request.
  - Do not download pasted inline images through `/open-apis/im/v1/images/{image_key}`. For this path, use the message resource API with the original `messageId`, `image_key`, and `type=image`.
- Current implementation:
  - Inline rich-text attachment parsing lives in [FeishuIncomingAttachmentParser.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuIncomingAttachmentParser.cs:1).
  - Direct `post` attachment submission logic lives in [FeishuChannelService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuChannelService.cs:1).
  - Normalization of Feishu rich-text content still lives in [FeishuPromptNormalizer.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuPromptNormalizer.cs:1).
- Search hints:
  - `MessageType=post`
  - `tag\":\"img`
  - `TryHandleInlinePostSubmissionAsync`
  - `[image]`

## Singleton channel services must not resolve scoped submission services from the root container

- Symptom:
  - A Feishu message path works in tests or partial flows, but runtime throws:
    - `Cannot resolve scoped service '...IMessageSubmissionService' from root provider.`
- Root cause:
  - `FeishuChannelService` is registered as a singleton.
  - `IMessageSubmissionService` and `IAttachmentStagingService` are scoped.
  - Calling `GetService<T>()` for those directly on the root provider is invalid under the default DI container.
- Working rule:
  - In singleton message/channel services, create a scope first, then resolve scoped services from `scope.ServiceProvider`.
  - If a fallback manual construction path is needed, also resolve its dependencies from that scope.
- Current implementation:
  - Inline post attachment submission now prepares the draft through a scoped helper in [FeishuChannelService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuChannelService.cs:710).
- Search hints:
  - `Cannot resolve scoped service`
  - `CreateScope()`
  - `PrepareMessageSubmissionAsync`
