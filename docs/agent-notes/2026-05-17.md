## Codex app-server transient reconnect errors must not terminate Feishu goal streams

- Symptom:
  - During Feishu `/goal` execution, the card can suddenly show a terminal error such as `閿欒锛歊econnecting... 1/5`.
  - A subsequent `/goal` status check then makes it look like the goal has stopped, even though the underlying Codex thread may still keep progressing server-side.
- Root cause:
  - `CodexAppServerSessionManager.HandleNotification(...)` previously treated every app-server `error` notification as terminal and immediately called `CompleteActiveOutput(...)`.
  - Codex app-server can emit transient transport errors like `Reconnecting... 1/5`, `Current interaction failed.`, or `An unknown error occurred.` while it is still retrying the active turn.
  - WebCode therefore closed the Feishu stream too early and surfaced a false terminal failure.
- Working rule:
  - Treat app-server reconnect-style `error` notifications as transient transport state, not as terminal turn failure.
  - Keep the active output stream open for those messages and wait for a real terminal signal such as `turn/completed`, `turn/failed`, process exit, or an explicit stop/pause/clear path.
  - Keep true non-transient `error` notifications terminal.
- Current implementation:
  - Transient error suppression now lives in [CodexAppServerSessionManager.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CodexAppServerSessionManager.cs:843).
  - Regression coverage lives in [CodexAppServerSessionManagerTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CodexAppServerSessionManagerTests.cs:140).
- Search hints:

## Syncing a cc-switch Provider for an active goal runtime must require Feishu confirmation before runtime reset

- Symptom:
  - A Feishu user can click `鍚屾 Provider` on a Codex session that is currently running in goal app-server mode.
  - The old implementation immediately started background sync, even if that session still had an active goal turn.
  - Because provider sync only updates the session-local snapshot and thread artifacts, users could wrongly assume the currently running goal had already switched to the new Provider.
- Root cause:
  - `HandleSyncSessionProviderAsync(...)` only checked whether the session tool was cc-switch managed.
  - It did not distinguish ordinary one-shot sessions from `UseGoalRuntime == true` sessions with an active turn or active Feishu execution.
  - `SyncCodexThreadProviderAsync(...)` updates the thread snapshot, but the already-running app-server process keeps its existing Provider/runtime state until WebCode explicitly resets that session runtime.
- Working rule:
  - For ordinary sessions, `鍚屾 Provider` can remain a fire-and-forget background action.
  - For goal-runtime sessions that still have an active execution or lingering app-server turn, Feishu must first show a confirmation card.
  - After explicit confirmation, WebCode should sync the thread Provider and then call `ResetSessionRuntimeAsync(sessionId, clearCliThreadId: false)` so the next goal turn restarts against the new Provider while preserving the CLI thread id.

## Session-manager card actions should refresh the current card in callback, not post a new card later

- Symptom:
  - `鍚屾 Provider` from the Feishu session-manager card used to return only a toast, then send a brand-new session-manager card later from a background task.
  - `鍒囨崲浼氳瘽` also returned only a toast, so the currently opened session-manager card still showed the old active session until the user reopened it.
- Root cause:
  - `HandleSyncSessionProviderAsync(...)` delegated the real work to a fire-and-forget background task and refreshed the card through `SendElementsCardToChatAsync(...)`, which always posts a new card message.
  - `HandleSwitchSessionAsync(...)` switched the active session in channel state, but never rebuilt and returned the updated session-manager card in the callback response.
  - For ordinary interactive cards, WebCode can reliably replace the current card only through the callback response `SetCard(...)`; there is no reusable current-card handle in this path like the streaming `cardId`.
- Working rule:
  - If a session-manager button is expected to mutate the session list or current-session marker immediately, do the work inside the callback and return `BuildCardActionResponseV2(...)` with the rebuilt manager card.
  - Avoid using `SendElementsCardToChatAsync(...)` for follow-up refreshes from those actions unless the intent is explicitly to create a new message card.
  - Preserve `show_all_sessions` in `switch_session` action payloads so an expanded manager card stays expanded after the refresh.
- Current implementation:
  - `switch_session` now rebuilds and returns the updated manager card in [FeishuCardActionService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/FeishuCardActionService.cs:2529).

## Goal runtime can be temporarily exited without clearing its CLI thread or resetting app-server state

- Symptom:
  - Users in a Feishu goal runtime session sometimes need to send ordinary supplemental chat messages without keeping that session in `UseGoalRuntime == true`.
  - Fully clearing or resetting the runtime would remove the easy later reactivation path for the same Codex thread.
- Working rule:
  - `临时退出补充会话内容` should only disable the session's `UseGoalRuntime` launch override.
  - Do not call `ResetSessionRuntimeAsync(...)`.
  - Do not call `StopSessionExecutionAsync(...)`.
  - Do not clear the stored CLI thread id.
  - Ordinary one-shot chat can keep using the same Codex thread, and goal runtime can later be reactivated explicitly.
- Card UX rule:
  - Streaming cards for goal runtime sessions should keep goal quick actions in three rows:
    - row 1: `/goal`, `/goal pause`
    - row 2: `/goal clear`, `/goal resume`
    - row 3: `临时退出补充会话内容`
  - Session-manager cards should use the same multi-row goal action layout for mobile readability.
  - Session-manager cards should hide the temporary-exit button when that goal runtime session still has an active turn; the click handler itself must still guard against active execution.
- Current implementation:
  - Temporary-exit action constant and button text live in [FeishuHelpCardAction.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Model/Channels/FeishuHelpCardAction.cs:17) and [GoalQuickActionDefaults.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Model/GoalQuickActionDefaults.cs:14).
  - Streaming-card goal action rows live in [GoalQuickActionCardHelper.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/Channels/GoalQuickActionCardHelper.cs:8).

## Re-entering goal runtime after temporary exit must resync the preserved Codex thread to the pinned session provider

- Symptom:
  - A Feishu user can temporarily exit goal runtime, send ordinary one-shot chat on the same session, then submit `/goal` again from the streaming card.
  - The new app-server process may fail during `thread/resume` with errors such as `failed to load configuration: Model provider ... not found`, even though one-shot `exec resume` still works on the preserved thread.
- Root cause:
  - Temporary exit intentionally preserves both `UseGoalRuntime == false` semantics and the existing `CliThreadId`.
  - When goal runtime was re-entered later, WebCode rebuilt the session-local Codex config snapshot but did not rewrite the preserved thread's rollout/sqlite provider metadata to match the session's pinned provider before `CodexAppServerSessionManager.EnsureThreadAsync(...)` called `thread/resume`.
  - App-server resume is stricter about stale provider ids than ordinary one-shot `exec resume`, so the preserved thread could become unrecoverable after provider drift.
- Working rule:
  - Before resuming an existing Codex thread into app-server goal runtime, do a best-effort thread-provider sync against the session's already pinned provider.
  - For Codex, prefer the actual `model_provider` name from the session snapshot or live `config.toml`; only fall back to `ChatSession.CcSwitchProviderId` when no TOML provider setting can be resolved.
  - Do not treat cc-switch `ActiveProviderId` or persisted `ChatSession.CcSwitchProviderId` as interchangeable with Codex thread metadata. Those values can be a provider record id like `abcredai-1776841713983`, while app-server `thread/resume` expects the TOML provider name such as `abcredai`.
  - Do not convert temporary exit into a destructive reset; keep it as a lightweight mode switch and only repair the preserved thread metadata on re-entry.
- Current implementation:
  - The re-entry sync hook lives in [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:2938).
  - The TOML provider-name resolver, pinned-provider resolution, and shared thread-sync core live in [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:3954), [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:4064), and [CliExecutorService.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain/Domain/Service/CliExecutorService.cs:4090).
  - Regression coverage lives in [CliExecutorServiceTests.cs](/D:/VSWorkshop/WebCode/WebCodeCli.Domain.Tests/CliExecutorServiceTests.cs:443).
- Search hints:

## Windows release packaging may need single-node publish with analyzers disabled to avoid Roslyn OOMs

- Symptom:
  - `dotnet build` or `dotnet publish` for the WebCode Windows release can fail on a busy workstation with Roslyn errors such as `Out of memory`, `Stack overflow`, or compiler crashes during `WebCodeCli.Domain`.
- Root cause:
  - The repository's release packaging path compiles a large `net10.0` solution with many warnings and big service files.
  - On a memory-pressured Windows desktop, the default publish configuration can let Roslyn/analyzers allocate enough state to fail before packaging completes.
- Working rule:
  - For the installer packaging path, force a single-node publish and disable analyzers, documentation generation, and debug symbols.
  - Keep this constraint scoped to release packaging; it is not a general code-quality recommendation for normal inner-loop builds.
- Current implementation:
  - The guarded publish flags now live in [build-windows-installer.ps1](/D:/VSWorkshop/WebCode/tools/build-windows-installer.ps1:450).
- Search hints:
  - `RunAnalyzers=false`
  - `DebugType=None`
  - `Out of memory`
