# WebCode 0.2.6 升级到 0.2.9 说明

## 适用范围

本文档面向从 `v0.2.6` 升级到 `v0.2.9` 的使用方、实施人员和运维人员。

建议升级路径：

- `v0.2.6 -> v0.2.9`

当前目标版本：

- `v0.2.9`

对应发布页：

- `https://github.com/lusile2024/WebCode/releases`

## 升级概览

从 `0.2.6` 到 `0.2.9`，整体变化可以归纳为 4 个方向：

1. `cc-switch` 被正式收口为受管助手的唯一 Provider 权威源。
2. Web、移动端、飞书三端的会话继续执行与流式交互能力增强。
3. 飞书附件消息链路补齐，支持图片、文件、待提交卡片，以及富文本粘贴图片直传 CLI。
4. Windows 发布包和 Reply TTS 运行时打包能力更完整。

## 分版本变化说明

### 一、0.2.7 相比 0.2.6 的主要变化

`0.2.7` 最核心的调整，是把 `cc-switch` 明确为受管助手的唯一 Provider 权威源。

涉及工具：

- `Claude Code`
- `Codex`
- `OpenCode`

主要行为变化：

- WebCode 不再依赖手工维护上述工具的 provider/profile/env 配置。
- Provider 的激活与切换，应统一在 `cc-switch` 中完成。
- 新会话在第一次执行时，会复制当前激活 Provider 的 live config，形成会话级快照。
- 旧会话不会因为 `cc-switch` 后续切换 Provider 而自动漂移。
- 只有显式执行“同步 Provider”时，旧会话才会切到当前激活 Provider。

同时，这一段还补了若干稳定性修正：

- 修复 Windows 下不同 Codex 安装路径导致的 CLI 解析问题。
- 改进 Feishu 中活动流式会话的识别与恢复。
- 修复 Codex 在 Feishu 补充消息场景下容易跑偏线程的问题。
- 让上游 CLI 错误能更明确地显示在 Feishu 侧。

升级影响：

- 如果你此前依赖手工修改 CLI 的全局 Provider 配置，升级后建议改成先在 `cc-switch` 切换，再在 WebCode 中同步会话。
- 旧会话行为会更像“终端窗口”，上下文稳定，但 Provider 不再自动跟随机器全局状态。

### 二、0.2.8 相比 0.2.7 的主要变化

`0.2.8` 主要增强了会话继续执行体验、飞书交互，以及 Reply TTS 能力。

#### 1. 会话继续执行与三端统一交互增强

主要变化：

- 补齐低打断 continue / native resume 能力。
- Web、移动端、飞书三端对同一会话的继续执行体验更统一。
- Feishu 流式卡片在切换、接管、补充消息、完成判定等场景下更稳定。
- 会话级模型/推理参数覆盖增强，不需要为了改单次行为去改全局默认值。
- 会话重命名、移动端抽屉、Feishu 卡片切换等场景更稳定。

#### 2. Codex `/goal` 与 Superpowers 工作流增强

主要变化：

- Codex `/goal` 快捷动作能力补齐。
- Superpowers 快捷动作与能力探测增强。
- 会话和卡片侧可以更方便地触发结构化工作流，而不只是发普通文本 prompt。

#### 3. Reply TTS 能力引入

`0.2.8` 这一段还引入了 Reply TTS 相关能力：

- 新增 Feishu Reply TTS 平台服务。
- 支持回复完成后排队执行 TTS。
- 支持 Feishu 音频上传与发送。
- 新增用户级 Reply TTS 设置持久化。
- 管理端增加 Reply TTS 控制入口。
- 集成本地 Kokoro / sherpa-onnx Reply TTS wrapper。

升级影响：

- Feishu 会话的“继续执行”“追加消息”“卡片接管”会比 `0.2.6` / `0.2.7` 更稳定。
- 如果启用 Codex，会明显受益于 `/goal`、resume、线程恢复相关改进。
- 如果计划启用飞书语音回复，可从该版本线开始评估 Reply TTS。

### 三、0.2.9 相比 0.2.8 的主要变化

`0.2.9` 最重要的新能力，是“统一附件提交链路”。

#### 1. Web / 移动端新增附件感知提交流程

主要变化：

- 聊天消息支持附件。
- 附件会进入统一的提交编排流程。
- 支持附件校验、路径归一化、暂存、历史持久化。
- 如果目标 CLI 支持原生附件，则优先走原生附加。
- 如果目标 CLI 不支持，则退化为工作区文件路径引用。

这意味着消息模型从“纯文本”升级为“文本 + 附件”的统一结构。

#### 2. 飞书顶层图片 / 文件消息走待提交卡片

如果用户在飞书直接发送：

- `image`
- `file`

现在会进入如下流程：

1. 下载飞书消息资源。
2. 暂存到当前会话工作区。
3. 回一张“待提交附件”卡片。
4. 用户补充说明。
5. 再把附件和说明一起提交给 CLI。

该流程统一适用于：

- 图片
- 文件

这解决了早期版本中“附件消息难以补充说明、直接转成无意义文本”的问题。

#### 3. 飞书富文本 `post` 粘贴图片 + 文本可直接送 CLI

飞书中“在文字输入框直接粘贴图片，再输入文字一起发送”的消息，并不是顶层 `image`，而是 `post` 富文本消息。

老版本问题：

- WebCode 只能看到类似 `[image]` 的占位文本。
- CLI 实际拿不到图片内容。

新版本行为：

- 从 `post` 富文本结构里解析内嵌 `img` 节点。
- 通过消息资源 API 把图片真实下载下来。
- 把图片和 accompanying text 一起走 prepared submission 提交给 CLI。

这使得下面这类场景真正可用：

- “这是什么图？”
- “帮我看这张截图报错”
- “检查这张 UI 截图哪里不对”

#### 4. 飞书附件和卡片链路稳定性修正

围绕附件链路，`0.2.9` 还补了很多细节修正：

- 草稿隔离与草稿清理。
- 附件归属限制到当前活跃会话。
- 避免旧草稿污染下一次执行。
- 稳定卡片提交按钮与回调链路。
- 修复单例服务错误解析 scoped 依赖导致的运行时异常。
- 加强 Codex 配置同步与 Feishu 附件草稿链路的稳定性。

#### 5. Windows 打包脚本修正

`0.2.9` 还修复了一个 Windows 发布流程中的关键问题：

- `tools/build-windows-installer.ps1` 之前会把 `publish/appsettings.json` 重写成非法 JSON。
- 现在改成了更稳定的 JSON 序列化方式，避免因为 CLI 注释字符串中包含嵌套引号而把配置文件写坏。

## Windows 发布包变化

从 `0.2.8` 到 `0.2.9`，Windows 发布包已经形成比较完整的发布资产集合：

- `WebCode-Setup-vX.Y.Z-win-x64.exe`
- `WebCode-vX.Y.Z-win-x64-portable.zip`
- `SHA256SUMS.txt`
- `RELEASE_NOTES.md`

发布包特征：

- `win-x64` 自包含，不要求目标机器单独安装 .NET Runtime。
- 安装版升级时保留已有 `appsettings.json`。
- 默认访问地址为 `http://localhost:6021`。
- 包含 Reply TTS wrapper 与相关运行时资源。
- 安装器会将 Reply TTS 相关模型、ffmpeg、Python runtime、venv 部署到可写的非系统盘目录。
- 如果机器只有系统盘可写，安装器会直接阻止安装。

## 升级影响与兼容性提醒

升级到 `0.2.9` 后，需要重点关注这些变化：

### 1. `cc-switch` 是唯一 Provider 权威源

对于受管助手：

- `Claude Code`
- `Codex`
- `OpenCode`

不建议再通过 WebCode 外的脚本、手工改配置文件、手工改 env 的方式直接切 provider。

推荐流程：

1. 在 `cc-switch` 中切换 provider。
2. 在 WebCode 中新建会话，或对旧会话执行“同步 Provider”。

### 2. 旧会话不会自动跟随机器当前 Provider

这是设计行为，不是 bug。

新会话与旧会话在 provider 行为上的区别：

- 新会话：第一次执行时拿当前激活 provider 快照。
- 旧会话：保持原快照，除非显式同步。

### 3. 飞书附件行为和旧版不同

顶层图片和文件消息现在会先进入“待提交附件”卡片流程，而不是直接执行。

这意味着：

- 用户会先看到一张待提交卡片。
- 需要在卡片中补充说明后再提交给 CLI。

### 4. 飞书粘贴图片 + 文字是特殊 `post` 流程

与顶层 `image` 不同，富文本 `post` 消息中的内嵌图片不会走待提交卡片，而是直接把图片和文字一起送给 CLI。

### 5. Windows 安装包对 Reply TTS 落地目录有要求

如果目标机器没有可写的非系统盘目录，安装器会停止，不继续安装 Reply TTS 运行资源。

## 建议的升级后验证清单

建议升级完成后，至少验证以下项目：

### 会话与 Provider 验证

1. 在 `cc-switch` 中切换一次 provider。
2. 在 WebCode 中新建会话，确认新会话拿到新的 provider。
3. 打开一个旧会话，确认旧会话没有自动漂移。
4. 手动执行“同步 Provider”，确认旧会话可以切到新 provider。

### 三端继续执行验证

1. 在桌面 Web 中启动一个会话。
2. 在移动端继续同一会话。
3. 在飞书中继续同一会话。
4. 确认三端上下文连续、线程不丢失。

### Codex 相关验证

1. 验证 Codex 会话 resume 是否正常。
2. 验证 `/goal` 快捷能力是否正常。
3. 验证 Feishu 补充消息是否仍然落在正确线程。

### 飞书附件能力验证

1. 在飞书直接发送一张图片，确认收到“待提交图片”卡片。
2. 在飞书直接发送一个文件，确认收到“待提交文件”卡片。
3. 在待提交卡片中补充说明并提交，确认 CLI 收到附件上下文。
4. 在飞书输入框中直接粘贴图片并输入一句话发送，确认图片和文字一起送到了 CLI，而不是只收到 `[image]` 占位符。

### Windows 发布包验证

1. 安装或解压最新 `v0.2.9` 发布包。
2. 启动后访问 `http://localhost:6021`。
3. 验证 `data/`、`logs/`、`workspaces/` 是否正常创建。
4. 如果启用 Reply TTS，验证相关资源是否正确部署。

## 升级建议

如果当前版本仍然是 `0.2.6`，建议直接升级到 `0.2.9`，不必逐个中间版本过渡。

推荐做法：

1. 备份当前 `appsettings.json`、数据库、工作区目录。
2. 升级到 `v0.2.9` 发布包。
3. 先验证 `cc-switch` provider 行为。
4. 再验证三端会话继续执行。
5. 最后验证飞书附件链路与 Reply TTS。
